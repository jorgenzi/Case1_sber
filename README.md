# AI-агент проверки целевого использования льготных кредитов

REST API сервис для автоматизированной проверки пакетов документов при заявлении льготных кредитов в банк. Система определяет типы документов по именам файлов, проверяет комплектность пакета в зависимости от программы (`federal` / `regional`), валидирует форматы и размеры файлов, и возвращает структурированный отчёт с итоговым статусом.

> **Статус:** MVP. Блок `extracted` (извлечение данных из документов) замокан — в продакшене планируется интеграция с OCR/AI-модулем.

---

## ✨ Возможности

- 📤 Загрузка пакета документов через `multipart/form-data`
- 🔍 Автоматическое определение типа документа по имени файла (договор, спецификация, счёт, акт/УПД)
- ✅ Проверка комплектности пакета для федеральной и региональной программ
- 📏 Валидация форматов (PDF, DOCX, JPG, PNG) и размера (до 20 МБ)
- ⚠️ Предупреждения для нераспознанных имён файлов
- 💾 Сохранение истории проверок в PostgreSQL
- 🐳 Запуск одной командой через Docker Compose

### Почему именно такой стек?

- **FastAPI вместо Flask/Django** — нативная поддержка async, автоматическая Swagger-документация, типизация через Pydantic (меньше бойлерплейта).
- **SQLAlchemy + Alembic вместо сырого SQL** — декларативные модели, версионируемые миграции, защита от SQL-инъекций.
- **PostgreSQL вместо SQLite** — production-ready, поддержка JSON для гибких полей (`issues`, `documents`), масштабируемость.
- **JSON-поля вместо отдельных таблиц для `issues`/`documents`/`extracted`** — для MVP это упрощает схему и избавляет от JOIN-ов. Проверка неизменяема после создания, поэтому денормализация оправдана. При росте требований (например, выборка по типу документа) `documents` можно вынести в отдельную таблицу.

## 🏗 Архитектура решения

Проект построен по принципу **слоёной архитектуры** с чётким разделением ответственности: доменная логика (`core`) ничего не знает ни про HTTP, ни про БД — это позволяет тестировать бизнес-правила напрямую, без моков и без поднятия Postgres.

```
HTTP-запрос
    │
    ▼
api/checks.py         тонкий роутер FastAPI:
    │                  парсинг формы/файлов, HTTP-статусы, коды ответа
    ▼
services/check_service.py    оркестратор: вызывает core, сохраняет результат
    │
    ├──▶ core/classifier.py, rules.py   чистые функции, без FastAPI/БД
    │
    └──▶ models/database.py, check.py   SQLAlchemy ORM, таблица checks
              │
              ▼
          PostgreSQL
```

**Путь одного запроса `POST /api/checks`:**

1. `api/checks.py` принимает `multipart/form-data` (`program` + файлы), валидирует HTTP-уровень (программа из допустимого списка, файлы не пусты), читает байты файлов.
2. `services/check_service.py` — оркестратор: для каждого файла вызывает `core/classifier.py` (определяет тип документа по имени), затем прогоняет `core/rules.py` (валидация каждого файла + проверка комплектности всего пакета + итоговый статус).
3. Результат упаковывается в ORM-объект `models/check.py` и сохраняется в БД.
4. `api/checks.py` конвертирует ORM-объект обратно в Pydantic-схему (`schemas/check.py`) — это уходит клиенту как JSON.

### Ключевые архитектурные решения

1. **Разделение `core` и `services`** — бизнес-правила (`rules.py`, `classifier.py`) не знают про БД и HTTP. Их можно тестировать изолированно, без моков.

2. **JSON-поля для `issues`/`documents`/`extracted`** — проверка является «снапшотом» на момент создания. Отдельные таблицы добавили бы JOIN-ы без выгоды.

3. **NFC-нормализация имён файлов** — macOS (Finder) сохраняет имена в NFD-форме (буква + диакритик раздельно). Без нормализации к NFC классификатор ломается на буквах `ё`, `й` — например, `счёт.pdf` не распознаётся как `invoice`.

4. **Матчинг по токенам через `startswith`** — защита от ложных срабатываний. Например, `"Счёт-фактура.jpg"` не определяется как `act`, хотя содержит подстроку `"акт"`.

5. **Порядок проверки типов** — `specification` проверяется раньше `contract`, чтобы `"Спецификация к договору.pdf"` корректно классифицировался.

6. **Поле `extracted` всегда пустое** — извлечение данных из документов (контрагент, сумма, дата, предмет договора) вынесено за скоуп этого сервиса и ожидает интеграции с отдельным AI/OCR-модулем. Схема ответа (`ExtractedSchema`) уже описывает форму этих данных, чтобы клиенты API могли на неё ориентироваться заранее.

7. **Время `checked_at` хранится в БД в UTC**, а в JSON-ответе API отдаётся в московском времени (`Europe/Moscow`) — конвертация происходит только на границе сериализации (`schemas/check.py`), внутренний код и БД по-прежнему работают в UTC.

## 📁 Структура проекта

```
app/
  core/       # чистая доменная логика — не импортирует ни FastAPI, ни SQLAlchemy
    classifier.py   # определение типа документа по имени файла
    rules.py         # правила комплектности пакета, валидация файла, статус
    config.py        # типизированные настройки из .env
  models/     # SQLAlchemy ORM
    database.py      # engine, сессии, Base, get_db() для Depends
    check.py         # таблица checks
  schemas/    # Pydantic-схемы запросов/ответов (контракт API)
  services/   # оркестрация: вызывает core, сохраняет результат через models
  api/        # FastAPI-роутеры — тонкий HTTP-слой поверх services
  main.py     # сборка приложения: FastAPI() + роутер + создание таблиц
alembic/      # версионируемые миграции схемы БД
tests/        # pytest — домен (classifier, rules), без БД и моков
```

## 📡 API

| Метод | Путь | Описание |
|---|---|---|
| `POST` | `/api/checks` | Загрузить пакет документов (`program` + `files`), получить результат проверки |

## 🚀 Быстрый старт

Рекомендуемый способ запуска — не требует локальной установки Python/Postgres.

### Предварительные требования

- Docker и Docker Compose (v2+)
- Git

### 1. Клонировать репозиторий

```bash
git clone <repo-url>
cd bank_docs_check
```

### 2. Создать файл `.env`

```bash
cp .env.example .env
```

### 3. Запустить проект

```bash
docker compose up --build
```

При первом запуске:
- скачаются образы PostgreSQL и Python (~300 МБ)
- создастся volume для данных БД
- применятся миграции Alembic (`alembic upgrade head`)
- запустится FastAPI на порту 8000

### 4. Проверить, что всё работает

- Swagger UI: **http://localhost:8000/docs**
- ReDoc: **http://localhost:8000/redoc**
- Сам `http://localhost:8000/` не обслуживается — в API нет отдельного маршрута на корень, используйте `/docs`

### 5. Остановить

```bash
# Остановить контейнеры (данные в БД сохранятся)
docker compose down

# Остановить и удалить данные БД (полный сброс)
docker compose down -v
```

## 🐍 Локальный запуск без Docker

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env  # поправить DATABASE_URL на localhost, если Postgres не в докере
alembic upgrade head
uvicorn app.main:app --reload
```

> **Важно:** зависимости (`psycopg2-binary`, `pydantic`/`pydantic-core`) закреплены под **Python 3.11** (см. `Dockerfile`). На более новых версиях Python (3.13+) сборка `psycopg2-binary` и `pydantic-core` из исходников может упасть — под них ещё нет готовых бинарных пакетов. Если локально доступен только Python 3.11 — используйте его для `venv`; иначе надёжнее запускать через Docker.

## ⚙️ Переменные окружения

Шаблон — `.env.example`, реальные значения — в `.env` (в `.gitignore`, в репозиторий не попадает):

```bash
cp .env.example .env
```

| Переменная | Назначение | По умолчанию |
|---|---|---|
| `DATABASE_URL` | Строка подключения SQLAlchemy к Postgres: `postgresql+psycopg2://<user>:<password>@<host>:<port>/<db>`. Хост `db` — имя сервиса Postgres в `docker-compose.yml`, работает только внутри Docker-сети. **При локальном запуске без Docker замените `db` на `localhost`.** | `postgresql+psycopg2://postgres:postgres@db:5432/bank_docs` |
| `MAX_FILE_SIZE_MB` | Максимальный размер одного загружаемого файла в мегабайтах. Файлы больше лимита отклоняются с `error`-issue в ответе `POST /api/checks` (см. `core/rules.py`). | `20` |

## 🧪 Как запустить тесты

Покрыт только домен (`app/core/classifier.py`, `app/core/rules.py`) — логика определения типа документа и формирования итогового статуса. Эти тесты не требуют БД и запускаются мгновенно, потому что домен намеренно вынесен из FastAPI/SQLAlchemy — тестируется напрямую, без моков.

### Вариант 1: В Docker (рекомендуется)

Не требует локальной установки Python — используется окружение уже поднятого контейнера `api`:

```bash
docker compose exec api pytest -v
```

### Вариант 2: Локально (без Docker)

```bash
# Создать виртуальное окружение
python -m venv venv
source venv/bin/activate  # Linux/macOS
# или: venv\Scripts\activate  # Windows

# Установить зависимости
pip install -r requirements.txt

# Запустить тесты
pytest -v
```

> **Важно:** зависимости (`psycopg2-binary`, `pydantic`/`pydantic-core`) закреплены под **Python 3.11** (см. `Dockerfile`). На более новых версиях Python (3.13+) сборка `psycopg2-binary` и `pydantic-core` из исходников может упасть — под них ещё нет готовых бинарных пакетов. Если локально доступен только Python 3.11 — используйте его для `venv`; иначе надёжнее вариант 1 (в Docker).
