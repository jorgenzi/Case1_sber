# AI-агент проверки целевого использования льготных кредитов

REST API сервис для автоматизированной проверки пакетов документов при заявлении льготных кредитов в банк. Определяет тип документа по имени файла, проверяет комплектность пакета для программы (`federal` / `regional`), валидирует формат и размер файлов, возвращает структурированный отчёт со статусом.

Статус: MVP. Блок `extracted` (извлечение данных из документов) замокан — в продакшене это будет отдельный OCR/AI-модуль.

## Возможности

- Загрузка пакета документов через `multipart/form-data`
- Определение типа документа по имени файла (договор, спецификация, счёт, акт/УПД)
- Проверка комплектности пакета для федеральной и региональной программы
- Валидация форматов (PDF, DOCX, JPG, PNG) и размера файла (до 20 МБ)
- Предупреждения для нераспознанных имён файлов
- История проверок в PostgreSQL
- Запуск одной командой через Docker Compose

## Стек

Python 3.11, FastAPI, SQLAlchemy 2.0, Alembic, PostgreSQL 16, Pydantic 2, pytest, Docker Compose.

FastAPI взяли ради async и автогенерации OpenAPI/Swagger из типов — меньше ручной документации. SQLAlchemy + Alembic вместо сырого SQL — декларативные модели и версионируемые миграции. Postgres вместо SQLite, потому что нужен нормальный JSON-тип для `issues`/`documents`/`extracted` и это всё равно ближе к прод-окружению.

`issues`/`documents`/`extracted` хранятся как JSON-поля, а не отдельные таблицы: проверка — это неизменяемый снапшот на момент создания, JOIN'ы тут не нужны. Если появится требование выбирать по типу документа — `documents` можно будет вынести отдельной таблицей.

## Архитектура

```
HTTP-запрос
    │
    ▼
api/checks.py                 тонкий роутер: парсинг формы/файлов, HTTP-статусы
    ▼
services/check_service.py     оркестратор: вызывает core, сохраняет результат
    ├─ core/classifier.py, rules.py   чистые функции, без FastAPI и БД
    └─ models/database.py, check.py  SQLAlchemy ORM, таблица checks
              ▼
          PostgreSQL
```

`core` не знает ни про HTTP, ни про БД — это позволяет тестировать бизнес-правила напрямую, без моков и без поднятого Postgres.

Путь запроса `POST /api/checks`: роутер принимает `program` + файлы, читает байты; `check_service` для каждого файла вызывает классификатор (тип по имени), потом `rules.py` (валидация файла + комплектность пакета + итоговый статус); результат сохраняется как ORM-объект и конвертируется обратно в Pydantic-схему для ответа.

Несколько решений, которые не очевидны из кода:

- NFC-нормализация имён файлов — macOS сохраняет имена в NFD (буква + диакритик раздельно), без нормализации `счёт.pdf` не распознаётся как invoice.
- Матчинг типа документа по токенам через `startswith`, а не по подстроке — иначе `"Счёт-фактура.jpg"` ложно матчится на `act`.
- `specification` проверяется раньше `contract`, чтобы `"Спецификация к договору.pdf"` не считалась договором.
- `checked_at` хранится в БД в UTC, в JSON-ответе отдаётся в московском времени — конвертация только на границе сериализации, БД и внутренний код по-прежнему в UTC.

## Структура проекта

```
app/
  core/       доменная логика — не импортирует FastAPI и SQLAlchemy
  models/     SQLAlchemy ORM
  schemas/    Pydantic-схемы запросов/ответов
  services/   оркестрация: core → сохранение через models
  api/        FastAPI-роутеры
  main.py     сборка приложения
alembic/      миграции
tests/        pytest — только домен, без БД и моков
```

## API

| Метод | Путь | Описание |
|---|---|---|
| POST | `/api/checks` | Загрузить пакет документов, получить результат проверки |
| GET | `/api/checks` | Список проверок (id, статус, дата, кол-во документов) |
| GET | `/api/checks/{check_id}` | Полная информация по одной проверке |

Swagger: http://localhost:8000/docs

## Запуск

```bash
git clone <repo-url>
cd bank_docs_check
cp .env.example .env
docker compose up --build
```

При первом запуске скачаются образы, применятся миграции Alembic, поднимется FastAPI на порту 8000.

Swagger — http://localhost:8000/docs, ReDoc — http://localhost:8000/redoc. Корень `/` не обслуживается, используйте `/docs`.

```bash
docker compose down      # остановить, данные в БД сохранятся
docker compose down -v   # остановить и снести volume с БД
```

### Без Docker

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env   # поправить DATABASE_URL на localhost
alembic upgrade head
uvicorn app.main:app --reload
```

Зависимости (`psycopg2-binary`, `pydantic-core`) закреплены под Python 3.11 — на 3.13+ их сборка из исходников может упасть, готовых wheel ещё нет. Если под рукой только Python 3.11 — используйте venv на нём, иначе надёжнее Docker.

## Переменные окружения

Шаблон в `.env.example`, копируется в `.env` (в `.gitignore`).

- `DATABASE_URL` — строка подключения SQLAlchemy к Postgres. Хост `db` — имя сервиса в docker-compose, работает только внутри Docker-сети; при локальном запуске замените на `localhost`.
- `MAX_FILE_SIZE_MB` — максимальный размер одного файла в МБ, по умолчанию 20. Файлы больше лимита отклоняются с error-issue в ответе.

## Тесты

Покрыт только домен (`core/classifier.py`, `core/rules.py`) — логика определения типа документа и итогового статуса. Без БД, работает мгновенно.

```bash
docker compose exec api pytest -v
```

Или локально — тот же venv, что и для запуска без Docker, дальше `pytest -v`.
